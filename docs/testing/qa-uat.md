# QA And UAT

This is the cross-surface release checklist.

Venue-specific details are also in `docs/venue-dashboard-uat.md`.

## Static Checks

```bash
npm run typecheck --workspaces --if-present
npm run lint --workspaces --if-present
npm run test --workspaces --if-present
npm run build --workspaces --if-present
flutter analyze
flutter test
```

Supabase:

```bash
supabase start
supabase db lint --local
psql "$SUPABASE_DB_URL" -f supabase/tests/bootstrap_required_objects.sql
psql "$SUPABASE_DB_URL" -f supabase/tests/rls_hardening_audit.sql
psql "$SUPABASE_DB_URL" -f supabase/tests/fet_wallet_reward_engine.sql
```

## Guest UAT

- Anonymous guest can open app/PWA.
- Guest can scan/open venue table QR.
- Venue/table context is resolved from secure QR token.
- Guest can view menu and item availability.
- Guest can place order.
- Guest sees off-platform payment guidance.
- Guest can open USSD or Revolut handoff when configured.
- Guest can track order service status.
- Guest receives FET when payment/reward conditions are met.
- Guest can view wallet ledger.
- Guest can create or join pool.
- Guest cannot join after lock/final match state.
- Winner receives settlement payout once.

## Venue UAT

- Owner, manager, and staff role checks pass.
- Orders queue shows live order details and payment/service status.
- Manual payment status changes write `payment_events` and `audit_logs`.
- Menu categories/items can be created, edited, hidden, and toggled.
- FET rewards can be configured by owner/manager only.
- QR table records are generated by backend token.
- Venue official pool uniqueness is enforced per match.
- Insights update from backend state.

## Admin UAT

- Admin sign-in works.
- Admin-only routes reject non-admin users.
- Match curation creates visible curated match options.
- Reward rules save with audit reason.
- Wallet freeze/adjustment requires reason.
- Pool operation queue shows settlement state.
- Settlement trigger is idempotent.
- Audit logs show admin and venue sensitive actions.

## Classified Marker Scan

Final hardening scan found no active unclassified `TODO`, `FIXME`, or `HACK` markers. Remaining "placeholder/mock/fake" terms are classified as test fakes, UI input placeholders, release docs that intentionally mention placeholder values, release metadata blockers, and loading placeholders.

Remaining `Math.random` hits are classified as non-security identifiers:

- request/log correlation ids;
- local UI notification ids;
- human-readable slug/order suffixes that are not authentication tokens.

Do not use `Math.random` for QR tokens, session tokens, wallet idempotency keys, pool settlement idempotency, or authentication.

Production paths must not depend on test fakes or sample data.
