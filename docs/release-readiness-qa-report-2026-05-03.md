# FANZONE Release Readiness QA Report

Date: 2026-05-03
Scope: Flutter mobile client, venue web PWA, admin web PWA, TV display PWA, Supabase migrations, Edge Functions, RLS/security posture, wallet/ledger, ordering, pools, dashboard, realtime, and TV/game requirements. `apps/website` is a separate public web PWA surface and must not be treated as the Flutter mobile client.

## 1. Executive Summary

Release recommendation: Not ready.

The implemented product has a working simplified sports-bar foundation: WhatsApp OTP tests exist, no-name mobile onboarding is present, venue browsing/ordering is implemented, off-platform payment handoff exists, staff manual payment confirmation is audited, user FET wallets use a ledgered mutation function, and basic pool staking/settlement flows exist.

Remediation update: the critical server-side product-rule gaps found in the first audit pass now have targeted Supabase hardening migrations and client/dashboard contract updates. The remote FANZONE Supabase project has migrations through `20260503150000_release_query_indexes.sql` applied. These add immutable unique fan IDs, fan-profile category limits, venue FET wallets and ledgering, venue-only pool creation, join-deadline enforcement, order-linked eligibility settlement, venue staff settlement wrapper, centralized game templates/sessions/questions, 20-question stable selection, first-correct answer protection, Music Bingo claims, venue-isolated TV screen state, hardened grants, security-advisor error remediation, targeted performance WARN cleanup for release-touched RLS policies, TV-safe live-question access, and release-critical order query indexes.

Authenticated UAT seed data is now present on the linked Supabase project: a dedicated UAT venue, admin/owner/manager/staff accounts, eligible/ineligible guests, orders, wallets, ledger rows, a venue-linked pool, a live game session with exactly 20 selected Fan Trivia questions, 100+ approved questions, teams, first-correct answer data, and TV screen state. Browser OTP login smokes now pass for the venue dashboard and admin PWA, and the standalone TV display renders the seeded live game question.

The app is still not signed off for production because production credential rotation remains mandatory after credential exposure in chat, Flutter mobile UAT must run on simulator/device rather than Flutter web/public PWA screenshots, full multi-device realtime UAT is still required across phone + venue dashboard + TV, and 224 Supabase performance WARN findings remain. Flutter, TypeScript, npm, remote SQL fixture checks, remote RLS tests, authenticated browser smokes, TV screenshots, and Supabase advisor ERROR checks pass in the current worktree/project.

## 3. Defect List

### FZ-QA-001 - Settlement pays winners without qualifying-order eligibility

- Severity: Blocker
- Area: Wallet / Pools / Settlement
- Steps: Create or join a pool with no paid order from the linked venue in the 2-hour pre-start window, choose the winning camp, settle the pool.
- Expected: Winner is marked `Won - Ineligible / No qualifying order`; no FET payout is credited.
- Actual: `settle_match_pool` counts all active result-camp entries as winners and credits `pool_win` to every winning entry. See `supabase/migrations/20260501155600_automated_pool_settlement_engine.sql:272` and `supabase/migrations/20260501155600_automated_pool_settlement_engine.sql:577`.
- Suggested fix: Add a server-side eligibility function that checks paid, non-cancelled orders from the linked venue between `start_time - interval '2 hours'` and start time. Use it inside settlement before any payout ledger rows are created.
- Status: Fixed on remote DB, pending live UAT.

### FZ-QA-002 - Entertainment games, question selection, first-correct scoring, Music Bingo, and TV display are not backend-backed

- Severity: Blocker
- Area: Games / Screen / Realtime
- Steps: Open venue portal `/games`, `/games/new`, `/games/:sessionId/control`, `/screen`, or inspect Supabase functions/migrations for game/session/question/bingo/screen tables.
- Expected: Central game templates, venue-linked sessions, 20 stored questions, one-answer-per-team, first-correct DB protection, bingo claims, team scoring, and realtime TV state.
- Actual before fix: Backend game/session/bingo/screen primitives existed, but authenticated host controls and TV display could not be proven because there was no seeded venue/staff/game fixture. TV live question reads also failed for anon display clients until grants were tightened.
- Suggested fix: Seed deterministic staff/game/screen fixtures, grant only TV-safe live-question reads, and smoke the browser display against the linked backend.
- Status: Backend, seed, TV grant, and TV layout fixed. Authenticated venue/admin browser smokes and TV live-question smoke pass; full realtime mode-switching UAT remains open.

### FZ-QA-003 - Venue/bar/platform FET wallets are missing

- Severity: Critical
- Area: Wallet / Dashboard / Pools / Games
- Steps: Inspect wallet schema and venue wallet pages.
- Expected: Separate user wallet, bar/venue wallet, and platform wallet where implemented, with all movements ledgered.
- Actual: `fet_wallets` is keyed by `user_id` only. Venue wallet pages explicitly say to show a real balance only when the wallet API is connected. See `supabase/migrations/20260423050000_sports_bar_clean_baseline.sql:10420` and `apps/venue-portal/src/features/target/TargetPages.tsx:252`.
- Suggested fix: Add owner-scoped wallets or a venue-wallet table with ledgered bar top-ups, bar stakes, game rewards, and platform movements.
- Status: Fixed on remote DB for venue wallets, pending UAT; platform wallet remains outside current MVP unless explicitly launched.

### FZ-QA-004 - Pools can be created without a linked venue

- Severity: Critical
- Area: Pools / Eligibility
- Steps: Create a mobile pool with default Global scope or call `create_pool` with `p_scope='global'`.
- Expected: Every prediction pool is linked to one venue/bar; user-created pools must select a linked bar.
- Actual: `create_pool` explicitly nulls `p_venue_id` for global and country scopes. Mobile pool creation defaults to global and only requires a venue for venue scope. See `supabase/migrations/20260501132000_pool_only_gameplay_controls.sql:214` and `lib/features/pools/screens/create_pool_screen.dart:33`.
- Suggested fix: Require `p_venue_id` for all user/bar pools and use country/global fields only as browsing filters.
- Status: Fixed on remote DB in the new `create_pool`/`create_match_pool` RPCs plus mobile/website create-pool UI; pending live UAT.

### FZ-QA-005 - Pool join backend accepts locked/live pools

- Severity: Major
- Area: Pools / Wallet
- Steps: Call `join_match_pool` for a pool whose status is `locked` or `live`.
- Expected: Joining is blocked after the deadline/lock; no wallet mutation occurs.
- Actual: Backend allows `open`, `locked`, and `live`. The shared UI helper only allows `open`, so client and backend rules conflict. See `supabase/migrations/20260501160000_pool_sharing_completion.sql:370` and `packages/core/src/pools.ts:64`.
- Suggested fix: Restrict backend join to `open` and enforce joining deadline/start time in the same locked transaction used for staking.
- Status: Fixed on remote DB in the overridden `join_match_pool` RPC; pending live UAT.

### FZ-QA-006 - Pool creation lacks required stake/deadline fields

- Severity: Major
- Area: Pools / Wallet
- Steps: Inspect `create_pool` RPC and venue pool creation.
- Expected: Linked bar, match, creator stake amount, participant stake amount, joining deadline/scheduled time, and bar wallet stake where relevant.
- Actual: `create_pool` accepts min/max stake and creator reward only; inserted `entry_fee_fet` is `0`; no creator stake or deadline is reserved. See `supabase/migrations/20260501132000_pool_only_gameplay_controls.sql:121` and `supabase/migrations/20260501132000_pool_only_gameplay_controls.sql:287`.
- Suggested fix: Add creator stake and participant fixed stake contract with ledgered reservation/refund semantics.
- Status: Partially fixed on remote DB, pending UAT. Participant fixed stake now maps to `entry_fee_fet`/`stake_min`; bar official stake is reserved from the venue wallet. User creator stake reservation remains a product decision if user-created pools must stake at creation time rather than on join.

## 4. Security/RLS Findings

Positive controls:

- Orders are RLS-protected for user/venue-member select, and direct client order updates are blocked by trigger. See `supabase/migrations/20260423050000_sports_bar_clean_baseline.sql:15472` and `supabase/migrations/20260423050000_sports_bar_clean_baseline.sql:9128`.
- User wallet and ledger reads are owner-scoped; wallet mutation primitive is service-role-only after hardening. See `supabase/migrations/20260423050000_sports_bar_clean_baseline.sql:14988` and `supabase/migrations/20260501162000_wallet_rpc_grant_hardening.sql:1`.
- `join_match_pool` uses a locked pool row and wallet transaction function, which protects basic overdraft/ledger behavior.
- New venue wallet mutation is exposed only through restricted RPCs and service/venue-manager checks; direct table writes are not granted to authenticated users.
- New game answer functions validate team membership and rely on DB uniqueness for one team answer and one first-correct answer per question.
- No dangerous DOM sink matches were found for `dangerouslySetInnerHTML`, direct `innerHTML`, `eval(`, or `javascript:` in the app/source scan.

Security gaps:

- Fan ID immutability, pool creation, join rules, settlement eligibility, venue wallet permissions, game answer authorization, and first-correct race protection are deployed and covered by remote SQL/RLS tests.
- Supabase advisor ERROR findings are clear after grant/view/RLS hardening. Performance WARN findings were reduced from 232 to 224 by fixing release-touched RLS init-plan policies; remaining WARNs are legacy RLS init-plan and multiple-policy findings across unrelated modules.
- Reviewer WhatsApp OTP support is time-bound for UAT and must be removed or rotated after review; it is not a production login shortcut.
- Production launch must wait for documented credential rotation.

## 5. Data Integrity Findings

- User wallet balance changes are generally ledgered through `wallet_post_transaction`, with idempotency and insufficient-balance checks.
- Venue wallet balance changes are ledgered through `venue_wallet_post_transaction`; direct balance mutation by users remains blocked by grants/RLS.
- Pool settlement now filters eligible/ineligible winners and refunds no-winner/cancelled pools in the hardening migration.
- Pool joining after lock/live is blocked by the overridden `join_match_pool`.
- Game answer duplicates and first-correct races are protected with DB constraints/indexes.
